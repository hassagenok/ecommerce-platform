# Объяснение, почему fix, update, misc плохо помогают команде:
Сообщения в commit по типу: fix, update, misc не помогают команде, потому что они не дают конкретного контекста об изменении и пути изменения. Из таких сообщения нельзя понять, где было изменена документация, исправлен ли баг, или кто-то сохранил случайно эксперимент. Из-за таких коммитов сложнее ориентироваться в истории и история заполняется лишним мусором.

# Три плохих сообщения и улучшенный вариант для каждого:
1. Плохое сообщение: fix. Улучшенный вариант: fix: fix Makefile dependency rules.
2. Плохое сообщение: document. Улучшенный вариант: docs: add lesson notes.
3. Плохое сообщение: update styles. Улучшенный вариант: style: update CSS file.

# Пояснение, почему message должен отражать смысл commit, а не только имя файла:
message должен отражать смысл commit для того, чтобы команде и reviewer было проще ориентироваться в истории по ключевым моментам и словам в message. Reviewer все равно будет читать diff, но правильное сообщение проложит путь к правильным мыслям изначально.

# Conventional Commits
Conventional Commits - это наш будущий стандарт коммитов. Пока важнее отличать плохой коммит от хорошо и понимать, в чем разница между хорошим коммитом и Conventional Commits. к примеру: fix Makefile dependency rules - хороший коммит,  fix: fix Makefile dependency rules - Conventional Commits.

# Локальная проверка перед commit:
make status
make diff
git diff --staged
make check

# Локальная проверка после commit:
git status --short
