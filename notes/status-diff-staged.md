# Вопрос на который отвечает git status:
Каково состояние рабочего каталога на данный момент?

# Вопрос на который отвечает git diff:
Какие изменения были внесены между рабочим каталогом, staging area и последним коммитом?

# Вопрос на который отвечает git diff --staged:
Какие изменения были подготовлены для будущего коммита, относительно последнего коммита

# Схема сравнения HEAD, staging area и working tree:
Рабочая среда изменений (working tree) - git add - подготовка изменений к следующему коммиту (staging area) - git commit -m - последний коммит, зафикмированное состояние (HEAD)

# объяснение, почему после git add обычный git diff может стать пустым:
git diff без аргументов сравнивает working tree с staging area, после git add изменения попадают в staging area, поэтому они перестают отличаться от working tree.

# короткий вывод, почему staged diff нужно читать перед commit.
Перед коммитом нужно читать staged diff потому-что это проверка будущего коммита, она покажет какие файлы и какие изменения были подготовлены для будущего коммита. Это помогает убедиться, что в коммит не попадёт лишний мусор

Перед коммитом делаем проверку:
make status - показал A  notes/status-diff-staged.md
make diff - пусто
git diff --staged - показал изменения в конкретном файле и показал само изменение
make check - показал A  notes/status-diff-staged.md

После коммита делаем проверку:
git status --short
