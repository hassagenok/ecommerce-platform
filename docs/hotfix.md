# incident context:
1. Symptom - симптом. Это то, что видит пользователь или система. Это проявление проблемы, но не её причина. Например: не открывается страница оплаты.
2. Affected version - версия приложения, в которой появилась проблема. Это помогает понять, какой релиз содержит ошибку. Например: v.2.12.4
3. Production line - ветка или линия разработки, которая используется для реальзый пользователей и где работает данная версия с поломкой. Например: main, production, release.

# hotfix flow:
hotfix flow - это порядок действий, по которому команда исправляет проблему в production.
Данный порядок действий выглядит так:
1. Обнаружение проблемы.
2. Создаем branch от production и названием hotfix/.... Не делать изменения напрямую в main или production.
3. minimal fix - делаем только необходимые изменения, чтобы устранить проблему, не добавляя новых функций и не меняя архитектуру.
4. make check - проверить, что все исправно работает. Провести все нужные тесты и другие автоматические проверки.
5. review - сделать быстрый PR/MR.
6. После успешных проверок и ревью вливаем обратно в production line.
7. Tag или release note - создать новый тег версии, например: v.2.12.5. Добавить release note с описанием исправления.

# follow-up:
После того как hotfix был исправлен, прошёл необходимые проверки и ревью и был влит в production line, нужно определить, в каких ещё линиях разработки может жить данный баг.
- Если команда использует develop, то фикс переносится через merge или cherry-pick.
- Если поддерживается release-branch, возможно нужен backport.
- Если есть support line для старой версии, нужен отдельный path release.

Главный вопрос не в том, какую команду лучше применить в данный момент, а в том: "Где ещё существует этот баг"
# direct work and force push

no direct work:
* Нельзя напрямую изменять main, production, или другую prodected line.
* Даже если это срочный hotfix, он должен пройти все процедуры обычной branch: new branch, pr/mr, checks.
* Нам нужно сохранить контроль изменений и не навредить prodected line.

no force push:
* Запрещаестя делать git force push в общие линии (main, production, release ...)
* Force push может перезаписать историю, удалить чужие коммиты и сделать сложным восстаноление после ошибки.
* hotfix должен быстро восстановить production line до исправного состояния, не нарушая безопасность и историю. Поэтому мы используем все стандартные процедуры даже на срочном hotfix.
