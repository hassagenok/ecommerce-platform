# protected main role:
main как protected line является не рабочей веткой, а линия, в которой сохраняется код, который прошёл все нужные проверки, review, PR/MR. В данной ветке сохраняются качественные изменения, которые считаются безопасными для проекта и не должны нарушать его стабильность.
Обычно такая ветка защищена такими правилами:
1. Direct push to main is disabled.
2. All changes go through PR/MR.
3. Required checks must pass.
4. Force push is disabled.
5. Merge decision belongs to maintainer, code owner or release owner.
Таким образом main выполняет роль стабильного проекта, а не пространства для разработки или экспериментов.

# scheme:
feature/* -> pull request -> review -> CI -> protected main

# Local self-check:
Локальная проверка это первый пункт в качественном коммите.
Перед тем как делать PR/MR, разработчик должен:
1. Собрать проект локально
2. Сделать все нужные тесты и получить хорошие результаты (make check, git status --short, git branch -vv, git log)
3. Убедиться в том, что изменения функционирует хорошо.

# Review:
Вторая часть проверки - это review, другой разработчик или команда проверяет изменения которые были внесены.
1. Поиск ошибок - выявление багов и уязвимостей до того, как они попадут в релиз.
2. Обмен знаниями - разработчики с меньшим опытом учаться у более опытных через review.
3. Единный стиль - Контроль того, чтобы код соотвествовал стандартам и был читаемым для всех.
4. Улучшение архитектуры - Предложение более оптимальных путей решении задачи.

# CI (Continuous Integration):
Автоматический gate качества.
1. commit - Отправка кода со стороны разработчика в систему контроля версий.
2. build - Специальный CI-сервер автоматически собирает проект в работающее приложение.
3. tests - Система прогоняет сотни или тысячи тестов, чтобы убедиться: новый код работает исправно и не сломал старые функции.
4. result - Команда получает мгновенный отчёт.

# запрет на direct push:
Запрет на direct push защищает от случайных или необдуманных изменений.
1.  Нельзя пушить напрямую в main
2. Любые изменения должны пройти PR/MR

# запрет на force push:
1. Запрещает переписывать историю main
2. Избегает изменения истории main и слом репозитория.

# maintainer или code owner.
Даёт права на merge определённым ролям, финальная точка проекта.
1. maintainer или code owner утверждают изменения и делают merge в main
2. на данных ролях весит отвестенность за стабильность ветки
3. контроль соблюдения правил CI, review, checks.

# Почему не работаем прямо в main:
Работа напрямую в main ломает модель доверия в команде.
От main могут синхронизироваться другие разработчики, небрежное использование main как рабочую ветку несёт последствия не только для тебя, а для всей команды.
1. Один неверный коммит может навредить всему проекту.
2. CI перестаёт быть гарантом качества
3. Откатывать становиться сложнее, дольше и дороже
4. Теряется понимание стабильности
5. Страдает вся команда

# Что считается ошибкой:
1. Сделать direct push (git push origin main) без PR/MR
2. Self-merge - автор сам делает approve + merge и этим убирает человеческий контроль качества.
3. merge при красном CI - тесты провалены, но изменения все равно мержится
4. Обход checks фразой "у меня локально работает" - окружения отличаются, поэтому локальная работа не гарантирует стабильность.
