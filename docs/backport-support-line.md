# difference between support line and develop:

support line - это старая ветка или серия веток разработки, где команда поддерживает старую версию ограниченным набором изменений, делается это по некоторым причинам:

1. Телефонная версия приложения может долго не обновляться
2. Клиенты могут оставаться на более старых версиях
3. enterprise контракт может требовать security fixes для предыдущей версии.
Поэтому у команды появляется support line.

Что важно: в support line не должны попадать новые изменения, большие структурные изменения поведения, которые не были обещаны клиентам этой версии.
Главная задача support line: получать важные фиксы с минимальным риском для поддержания исправности старой версии приложения.

Разница между develop существенная. Develop - это линия разработки с будущими изменениями, которые ещё не видны обычным пользователям, в этой линии готовятся features для новой версии production. По этой же причине нельзя вливать фикс из develop, даже если он может пофиксить support line, ведь можно случайно вместе с фиксом попасть и новое изменение или конфликт.

# backport decision:
backport decision - это решение о том, нужно ли переносить изменение или фикс в старую поддерживаемую ветку.
* Affected version - указывает на то, какие версии имеют эту проблему.
* Source fix - исходное исправление, указывает на то, откуда было взято исправление.
* Target support line - указывает на то, куда переносится исправление.
* Risk - указывает на риски, которые могут принести изменения.
* Reasons - причина, по которой были внесены исправления.
* Scope - объем изменений.
* Checks - набор проверок.
* Release - информация о том, какой релиз будет содержать этот backport.

# safe command flow:
Safe command flow для backport - это последовательность действий, чтобы перенести фикс в support line безопасно, с сохранением всей истории и проверкой совместимости.
Последовательность действия:
1. Создаем новую backport ветку
2. Переносим source fix через cherry-pick
3. Если возникнут конфликты - решаем их
4. Запускаем проверки для support line
5. Создаём PR/MR
6. Review
7. Подготавливаем patch release note

# Почему нельзя просто merge develop в support line:
develop может хранить нужный нам fix для support line, но до него были внесены features (новая библиотека, новые функции, рефакторинг). Если мы сделаем merge develop, то наша support line получит ряд новых изменений, которых не ожидали пользователи.
Поэтому используют backport, чтобы перенести конкретный фикс в support line, без добалвения новый изменений, о которых могли не знать пользователи.
